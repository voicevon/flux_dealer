# 芦笋分拣机迷你控制器 — 需求规格说明书 (SRS)

> **文档版本**: v1.0  
> **日期**: 2026-05-18  
> **状态**: 正式发布  

---

## 1. 引言与背景

本系统是多轴同步芦笋物理分拣机构的核心控制大脑。芦笋通过高频输送带被逐根抛落至分拣机入口。为了将不同品级（根据手机端 `sorter_mini_phone` 摄像头视觉主控实时判定）的芦笋精确引导至对应的物理通道，控制器必须极速响应落料信号，推进多级流水线节拍，同步控制多路（当前为 8 路，未来可级联扩展至 16 路）翻转机构完成分选动作。

---

## 2. 业务场景与用例模型 (Use Cases)

```
                     ┌──────────────────┐
                     │ sorter_mini_phone│ (视觉主控/手机端)
                     └────────┬─────────┘
                              │
                              │ BLE 下发 Target ID
                              ▼
┌──────────────┐       ┌──────────────┐
│  入料传感器  ├──────►│   控制器MCU  │ (实时状态调度机)
└──────────────┘       └──────┬───────┘
  (物理落料检测)               │
                              │ 脉冲/屏蔽/移位总线控制
                              ▼
                       ┌──────────────┐
                       │   分拣机构   │ (电机协同阵列)
                       └──────────────┘
```

### 2.1 正常分拣核心用例 (Normal Sorting Loop)
*   **触发条件**：光电限位传感器检测到芦笋滑入入口（检测到边缘电平变化）。
*   **基本流 (Basic Flow)**：
    1.  控制器产生“新物料落入”节拍信号。
    2.  控制器从输入队列中提取手机主控端提前下发的 `Target ID`（品级分类，例如出口 1~8）。
    3.  控制器更新多级物理流水线的镜像缓存，将新 Target ID 推入第一级，其余历史物料依次推移至下一级。
    4.  控制器计算本节拍中需要同步运作的电机动作，并向对应的电机驱动器输出同步脉冲和正反转方向。
    5.  多级翻转机构同步旋转 90°，在极短的物理阻滞前将各级物料精准导向下一级分支或剔除槽。
    6.  物料动作完毕后，电机恢复锁步，延时滑行结束，状态机复归，等待下一次落料触发。

### 2.2 自动寻零用例 (Auto Homing)
*   **触发条件**：系统上电、复位或接收到手机端下发的 `COMMAND_HOMING` 指令。
*   **基本流 (Basic Flow)**：
    1.  系统清空一切运行中队列 and 流水线缓存。
    2.  系统控制全轴电机同时反向运行，寻找绝对物理零点限位开关。
    3.  各轴碰到物理限位后，自动刹车停转，完成整机的物理绝对零点校准。

### 2.3 异常保护用例 (Error & Fault Protection)
*   **用例 1：寻零超时保护**  
    若归零过程中发生限位开关物理断线或卡死，系统必须在设定时间窗口（10 秒）内强制中止运动，切断电机使能，上报 BLE 错误码。
*   **用例 2：通信断联保护**  
    当主控手机与控制器蓝牙断开时，系统应停止新物料接收并安全锁死，防止在无主控状态下发生物理机械碰撞或漏分拣。

## 3. 非功能性需求 (Non-Functional Requirements)

*   **实时响应性**：  
    物料入口传感器的检测判定时延必须控制在毫秒级，防止由于反应迟缓导致芦笋错落过分拣板。
*   **抗干扰防抖**：  
    物理落料过程中存在环境光线干扰与物料边缘机械弹跳，传感器必须具备强大的软件抗抖动过滤能力。
*   **通信可靠性**：  
    蓝牙通信必须使用高抗干扰协议栈，在潮湿、电磁辐射大的农业分拣车间内必须稳定保持高吞吐连接。
*   **多机共存安全防护**：  
    车间中并存多台分拣机时，**绝对禁止交叉连接**，必须具备 100% 物理一对一绑定的配对方案。
*   **架构可伸缩性**：  
    未来硬件在**不更换 ESP32 主 CPU** 的前提下，软件应能低成本地从 8 路无缝扩充到 16 路。
