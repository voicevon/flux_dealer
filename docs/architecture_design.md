## 1. 软件设计原则

为了保证多路高频控制下的系统稳定性与代码可维护性，软件总体架构遵循以下核心原则：

*   **单一职责原则 (SRP)**：
    系统将控制逻辑的各个维度划分为高度内聚的独立类。`SorterController` 仅负责整体状态机的转移与流水线节拍的推进；`MotionPlanner` 专门执行数据驱动的路由策略计算；`HomingController` 专门管理各轴的多阶段归零规约；硬件抽象层 (HAL) 的各组件则完全内化引脚操作与移位总线读写。
*   **硬件与逻辑完全解耦 (Hardware-Logic Decoupling)**：
    以 `MotionPlanner` 为代表的业务决策层在代码设计上**没有任何 Arduino API 或硬件寄存器依赖**。这使得分拣决策算法（如路由表推演）能够完美地脱离物理板卡，在 Native PC（如 GCC 环境）中进行 100% 覆盖率的单元测试，极大地加速了逻辑验证并提高了重构的安全性。
*   **高吞吐量并发流水线模型 (Concurrent Beat Pipeline)**：
    软件采用内部数组 `_pipeline[]` 镜像物理分拣机上的多级滑道。当新节拍到来时，数组整体串行推移，允许同一时间有多根芦笋在分拣机滑道中由不同级的电机同步、协同输送，大幅提升物理分拣机构的吞吐效率。

## 2. 总体软件分层架构

分拣机控制器软件架构采用严密的分层设计，以隔离变化，自顶向下分为五个核心层次：

### 2.1 分层架构模型

```
┌────────────────────────────────────────────────────────┐
│                      1. 应用层 (App)                    │
│      main.cpp (系统入口、看门狗喂狗、主事件循环调度)       │
└──────────────────────────┬─────────────────────────────┘
                           ▼
┌────────────────────────────────────────────────────────┐
│                  2. 核心调度与状态机层                  │
│   BleManager (蓝牙通信)   ◄───►   SorterController FSM   │
│  (处理底层事件/上报错误)          (流水线节拍状态转换控制)  │
└──────┬───────────────────────────────────┬─────────────┘
       │                                   │
       ▼                                   ▼
┌──────────────────────────────┐   ┌─────────────────────┐
│    3. 策略与运动规划层 (Logic) │   │     输入队列层       │
│  MotionPlanner (静态路由表)  │   │  InputQueue (FIFO)  │
│  HomingController (归零控制)  │   │ (线程安全环形缓冲区) │
└──────┬───────────────────────┘   └───────────┬─────────┘
       │                                       │
       └───────────────────┬───────────────────┘
                           ▼
┌────────────────────────────────────────────────────────┐
│                 4. 硬件抽象层 (HAL / Driver)            │
│  MotorHardware     │  ShiftRegisterBus │  SensorInput  │
│ (共用STEP/屏蔽EN)  │ (74HC595/165 SPI) │ (光电开关防抖) │
└──────────────────────────┬─────────────────────────────┘
                           ▼
┌────────────────────────────────────────────────────────┐
│                      5. 物理硬件层                      │
│     ESP32 MCU  │  HC595/HC165  │  A4988 步进驱动 ➔ 电机 │
└────────────────────────────────────────────────────────┘
```

### 2.2 模块控制流向

1.  **输入控制流**：
    *   **蓝牙数据入队**：手机端通过 BLE 下发 Target ID 触发 ESP32 蓝牙中断，`BleManager` 截获数据，执行临界区原子锁定后，将 ID 推入环形缓冲区 `InputQueue`。
    *   **传感器落料触发**：光电开关传感器触发边缘检测，`SensorInput` 执行防抖处理，若检测到芦笋落入入口（上升沿），向核心调度状态机发送“新节拍就绪”信号。
2.  **调度环流**：
    *   `SorterController` 监听到新节拍信号，从 `IDLE` 转移至 `NEW_BEAT_PREP`。
    *   流水线向前移位，`InputQueue` 弹出队首目标，填入第一级流水线 `_pipeline[0]`。
    *   核心调度器将当前流水线数组传入 `MotionPlanner::planGroupMove()`。
3.  **规划与执行环流**：
    *   `MotionPlanner` 检索静态只读的二维路由表 `ROUTING_TABLE`，根据各级当前的物品目标和电机的原始装配方向，计算出本节拍中：**哪几个电机需要使能、哪几个电机需要屏蔽、各参与电机的正反转方向（HC595 需要的 Bitmask）**，包装为 `MotionCommand` 结构体回传。
    *   `SorterController` 解析该命令，调用 `ShiftRegisterBus` 在模拟 SPI 总线上锁存方向数据，调用 `MotorHardware` 设置对应的 EN 屏蔽掩码，并指定共享 STEP 发生器进行 90° 行程的加减速运动。
    *   步进电机开始同步运转。状态机转移至 `MOVING_TO_ROUTE` 挂起等待。
    *   运动完成后，滑行延时（`SLIDE_WAIT_MS`）保证芦笋借助重力滑入下一级，然后状态机复归 `IDLE` 挂起，等待下一个落料沿。

## 3. 核心状态机 (SorterFSM) 设计

系统的业务主循环由 `SorterController` 类的有限状态机 (FSM) 驱动。通过精确管理传感器事件、滑行定时器与电机运动状态，实现与分拣物理进程的高度同步。

### 3.1 状态转移矩阵 (State Transition Table)

| 源状态 | 触发事件/条件 | 转移目标状态 | 执行动作 |
| :--- | :--- | :--- | :--- |
| `IDLE` | 入口光电传感器触发上升沿 | `NEW_BEAT_PREP` | 读取传感器沿状态，准备流转 |
| `NEW_BEAT_PREP` | 无（单步即时转移） | `MOVING_TO_ROUTE` | 流水线数组移位，出队 Target ID；调用 `MotionPlanner` 生成控制指令；向 HAL 写入 DIR 与 EN 屏蔽字；启动 `sharedStepper` 恒加速度步进 |
| `MOVING_TO_ROUTE`| 电机物理到位 `!_motorHardware.isMoving()` | `SLIDING_WAIT` | 物理位置重置 `setCurrentPosition(0)` 防止数值溢出；锁定当前时间戳 `_timerMark = millis()` 进入落料等待 |
| `SLIDING_WAIT` | 滑行时间到 `millis() - _timerMark >= SLIDE_WAIT_MS` | `COMPLETED_BEAT` | 逻辑判定物品已滑入下一级，本节拍物理完成 |
| `COMPLETED_BEAT` | 无（单步即时转移） | `IDLE` | 重置节拍，等待下一次光电信号触发 |
| *任意状态* | 接收到 BLE 归零命令 / 触发 `triggerHoming()` | `HOMING` | 清空 `InputQueue`；清空流水线 `_pipeline`；重置 `HomingController` 状态机并发起寻零 |
| `HOMING` | 所有轴限位触发且返回确认 | `IDLE` | 清除归零状态，系统就绪 |
| `HOMING` | 寻零时间超时 (超出安全窗口) | `ERROR_STATE` | 标记错误码 `ERR_HOMING_TIMEOUT`；关闭总脉冲屏蔽所有电机；上报 BLE 错误 |
| `ERROR_STATE` | 接收到 BLE 软复位命令 / 触发 `clearError()` | `HOMING` | 清除异常错误码，重新启动自动寻零校准 |

### 3.2 异常挂起与恢复设计
*   **硬件保护挂起**：一旦进入 `ERROR_STATE`，主循环将挂起一切流水线运作，同时硬件上通过将 8 路 EN 屏蔽引脚拉高，强制所有步进驱动屏蔽脉冲并进入完全静态通电锁步（或断电保护）状态，等待上位机通过蓝牙下发 `COMMAND_RESET` 指令执行软复位和重新归零。
*   **看门狗保护机制**：主循环运行 ESP32 硬件看门狗 (`esp_task_wdt`)，超时阈值设为 10 秒。任何状态机卡死（如阻断性死循环或硬件总线死锁）都将导致看门狗超时，触发 MCU 硬件硬复位以确保设备无人值守下的绝对安全。

## 4. 逻辑运动规划与查表规划 (MotionPlanner)

`MotionPlanner` 承担系统的核心算法规划任务。其本质是将高级的**流水线状态数组**，通过查表与位运算转换为底层硬件能够直接消费的**低级控制指令位图**。

### 4.1 核心数据结构 `MotionCommand`

`MotionPlanner` 不控制任何物理引脚，它只输出一个逻辑结构体：

```cpp
struct MotionCommand {
    uint8_t enable_mask;  // 8位屏蔽字掩码 (1 = 运动放行, 0 = 使能屏蔽锁步)
    uint8_t dir_state;    // 8位方向状态位图 (1 = CW 正转, 0 = CCW 反转)
    bool    move_any;     // 标记本节拍是否需要执行物理步进 (是否有任何电机激活)
    long    steps;        // 本次运动的统一步数值 (ROTATE_LEFT_POS 或 ROTATE_RIGHT_POS)
};
```

### 4.2 路由查表决策逻辑
系统维护一个二维静态矩阵 `ROUTING_TABLE[MAX_TARGETS + 1][NUM_MOTORS]`。该矩阵以目标 Target ID（1 至 8，0 为空槽）为行索引，以电机轴号（M0 至 M7）为列索引：
*   **动作定义**：
    *   `+1`：正转（向前推进，物料滑向下一级分支）。
    *   `-1`：反转（剔除，物料在此级直接滑落进对应的终端物理槽位）。
    *   `0`：无动作。

根据当前各流水级上的物料 Target ID，规划器循环判定每个电机的行为，并结合电机的物理装配朝向映射，生成本节拍的全局并行控制命令。此查表映射为纯逻辑计算，不依赖任何硬件，也是单元测试验证的绝对重点。

## 5. 多线程与并发通信安全设计

系统运行于双核 ESP32 芯片上，通信层与核心调度业务层处于不同的线程边界，因此线程安全与并发控制是软件架构高可用性的生命线。

### 5.1 线程边界与并发环境

*   **BLE 协议栈线程**：
    `BleManager`（基于 NimBLE 蓝牙协议栈）运行于 FreeRTOS 的高优先级后台任务线程。当手机端 `sorter_mini_phone` 下发 Target ID 时，会直接触发蓝牙接收中断，并在 BLE 任务线程的上下文中回调 `onWrite()` 写入方法。
*   **主循环线程**：
    `main.cpp` 中的 `loop()` 运行于标准的 FreeRTOS 循环任务中，以非阻塞形式循环调用 `SorterController::update()` 状态机调度。
*   **数据隔离桥梁**：
    `InputQueue` 是隔离这两个线程、避免全局竞争的唯一数据管道。

### 5.2 基于 ESP32 临界区的原子性防护

由于 NimBLE 线程可以抢占主循环，如果对队列的读写（更新环形缓冲指针）在非原子状态下被中断，将导致指针损坏或数据污染。系统在 `InputQueue` 中实现了严格的无锁/临界区双重保障：
1.  **指针 `volatile` 约束**：
    队列的头指针 `_head`、尾指针 `_tail` 以及缓冲数组均添加 `volatile` 强修饰，强制编译器每次直接从物理 RAM 读写，杜绝因 CPU 寄存器缓存或编译器激进优化导致的“脏读/脏写”。
2.  **FreeRTOS 临界区互斥 (Spinlock)**：
    在 `push()`（蓝牙写入侧）与 `pop()`（主循环读取侧）中，使用 ESP32 SDK 专属的 `portENTER_CRITICAL()` 和 `portEXIT_CRITICAL()` 进行原子级包裹。
    *   在进入临界区后，系统将**强行关闭当前 CPU 核的硬件中断与任务调度**，防止读写缓冲区时被任何抢占式中断打断。
    *   执行极速的数据推入/弹出和指针回绕更新，随后立即退出临界区恢复中断，确保总线抖动和高频蓝牙写入时的绝对线程安全与极小锁开销。
